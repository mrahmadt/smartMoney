# SMS Parsing Pipeline

The SMS parsing pipeline is the core engine of SmartMoney. It receives raw bank SMS messages, extracts structured transaction data, and creates transactions in Firefly III.

## How It Works

### 1. SMS Reception

Bank SMS messages arrive via a webhook at `POST /api/sms/filter`. The `SMSController` handles the incoming request:

- **Sender validation:** Checks if the SMS sender (bank name) is registered and active in the `sms_senders` table
- **Duplicate detection:** Generates an MD5 hash of the message content and checks against `message_hash` to prevent duplicate processing
- **Minimum length check:** SMS shorter than `parsesms_min_sms_length` (default: 30 chars) are rejected
- **Invalid transaction filtering:** Checks against keyword patterns (OTP, passwords, activation codes) to skip non-transactional SMS

If valid, the SMS is stored in the `smses` table and a `parseSMSJob` is dispatched to the queue.

### 2. SMS Cleaning

Before parsing, the SMS text is preprocessed:

- **Keyword removal:** The `Keyword` model stores patterns for phone numbers, passcodes, dates, URLs, and other noise. These are stripped before parsing.
- **Hidden character removal:** Non-printable characters are removed
- **Whitespace normalization:** Extra spaces and newlines are collapsed

### 3. Regex Parsing (Primary)

When `parsesms_regex_enabled` is `true` (default), the system first tries regex-based extraction:

- `SMSRegularExp::findInvalidRegExp()` — checks if any "invalid" regex matches (e.g., balance notifications, marketing SMS). If matched, the SMS is rejected.
- `SMSRegularExp::findValidRegExp()` — tries all active regex patterns for the SMS sender. Patterns use PHP named groups to extract:
  - `amount`, `currency`
  - `transactionDateTime`
  - `MyAccountNumber`
  - `OtherAccountName`, `OtherAccountNumber`
  - `fees`, `feesCurrency`
  - `totalAmount`, `totalAmountCurrency`

Regex patterns are stored per-sender and set to active/inactive. New patterns are automatically generated by the AI agent (see below).

### 4. AI Parsing (Fallback)

When regex matching fails and `parsesms_failback_ai` is `true`, the system calls the `parseSMS` AI agent:

- **Model:** Configurable via `parsesms_model` setting (default: `gpt-5-mini`)
- **Input:** Raw SMS text
- **Output:** Structured JSON with all transaction fields plus a PHP regex pattern
- **Regex learning:** If `parsesms_regex_enabled` is true, the AI-generated regex is saved via `SMSRegularExp::storeRegularExp()` for future use. This means the system learns from each new SMS format — the AI is consulted once, and subsequent messages of the same format are parsed by regex.

### 5. Category Detection

After transaction data is extracted, the category is determined:

1. **Withdrawal** → automatically categorized as "Cash"
2. **Transfer** → automatically categorized as "Transfer"
3. **Payment/Deposit** → looks up the merchant name in `CategoryMapping`:
   - If found: uses the mapped category
   - If not found and `parsesms_failback_detect_category_ai` is enabled: calls the `SMSCategory` AI agent
   - The AI result is stored in `CategoryMapping` for future lookups

### 6. Transaction Creation

The `Transaction::createTransaction()` method:

- Finds the local `Account` by matching the SMS sender and account shortcode
- Resolves the Firefly III `source_id` from the account
- Gets the `budget_id` from the account
- Handles multi-currency transactions (converts fees and amounts using Firefly III exchange rates)
- Calls the Firefly III API to create the transaction
- Sets tags (`regex:{id}` or `by AI`) for traceability

### 7. Post-Transaction Processing

After a successful transaction creation:

- **Alert:** Creates a transaction notification via `Alert::newTransaction()`
- **Pending category review:** If the merchant's `CategoryMapping` has alternative categories, creates a `PendingCategoryReview` record
- **Abnormal destination check:** Compares the transaction amount against the historical average for the same merchant
- **Unusual frequency check:** Detects if the same category has an unusual number of transactions today

## Key Files

| File | Purpose |
|------|---------|
| `app/Http/Controllers/SMSController.php` | Webhook endpoint |
| `app/Jobs/parseSMSJob.php` | Main processing pipeline |
| `app/Models/SMS.php` | SMS storage and validation |
| `app/Models/SMSSender.php` | Registered bank senders |
| `app/Models/SMSRegularExp.php` | Regex patterns per sender |
| `app/Models/Keyword.php` | SMS cleaning rules |
| `app/Models/ParseSMS.php` | LLM parsing and category detection |
| `app/Models/Transaction.php` | Firefly III transaction creation |
| `app/Ai/Agents/parseSMS.php` | AI SMS parser agent |

## Settings

| Key | Default | Description |
|-----|---------|-------------|
| `parsesms_enabled` | `true` | Master switch for SMS processing |
| `parsesms_regex_enabled` | `true` | Enable regex-based parsing |
| `parsesms_failback_ai` | `true` | Use AI when regex fails |
| `parsesms_failback_detect_category_ai` | `true` | Use AI for category detection |
| `parsesms_model` | `gpt-5-mini` | LLM model for SMS parsing |
| `parsesms_min_sms_length` | `30` | Minimum SMS length to process |
| `parsesms_store_invalid_sms` | `true` | Store invalid SMS for debugging |

## Example Flow

```
Input SMS: "Purchase Amount: SAR 45.00 At: STARBUCKS Account: ***7632 Date: 2026-03-15 14:30"

1. SMSController receives via POST /api/sms/filter
2. Sender "ALRAJHI" is active → passes validation
3. Message hash is unique → not a duplicate
4. parseSMSJob dispatched
5. Regex matches: amount=45.00, currency=SAR, merchant=STARBUCKS, account=***7632
6. CategoryMapping lookup: STARBUCKS → "Cafe"
7. Account lookup: sender=ALRAJHI, shortcode=7632 → firefly_account_id=5
8. Firefly III API: POST transaction (withdrawal, 45 SAR, STARBUCKS, Cafe)
9. Alert created: "Spent 45 SAR"
10. Abnormal check: 45 SAR is normal for STARBUCKS (average: 40)
```
